# 条款精准匹配改造设计稿（草稿 v0.1）

> 目标：把审核从「AI 自己翻整本规范」改造成「代码先把条款撕下来，摊在 AI 面前」。
> 本稿不改代码，只定流程、数据结构、改动点和实施顺序。

---

## 0. 朋友那套逻辑，我的理解（一句话版）

> **范式转换：从「AI 主导检索」改成「代码主导检索 + AI 主导判断」。**

朋友说的三个 hook，本质是在审核流水线中间，插进三个**确定性函数**（不是 React hook、不是中间件），把原来「软、会漏」的 AI 翻书环节，换成「硬、一定捞到」的 if 规则。

为什么是规则匹配而不是向量检索（embedding）？——因为建筑规范是**强结构化**的：规范号、条款号都是确定的，`DB44/T 1876-2016 第 5.2.3 条` 就是那一条，没有「相似度」。规则匹配比模糊向量更准、更可解释、更不会幻觉。

核心理念贯穿全文：**不让 AI 自己翻书，先把那一页撕下来放在它面前。**

## 1. 现状诊断（基于真实代码 + 真实目录）

| 环节 | 现状函数 | 问题 |
|------|---------|------|
| 专业识别 | `identifyProfessionTypes` | 关键词打分（`matches.length * keyword.length`），只分到 13 个**大类**，分不出**构造类型**（轮扣式 / 盘扣式 / 扣件式） |
| 文件匹配 | `matchRelevantFiles` | `file.filePath.includes(folderName)` —— 把整个专业文件夹全捞 |
| 上下文裁剪 | `extractKnowledgeContext` | 每个文件截断到 25KB，整本丢给 AI |
| 危大判定 | （无） | 目录里有 `2危大工程专项方案 / 3一般专项安全方案` 这层，代码完全没用上 |

**真实数据：**
- 知识库共 **574 个 md 文件**。
- 装配式一个专业 **121 本**，模板 41 本，钢结构 50 本。
- 现在的做法 = 把一个专业几十本规范整本塞给 AI，再各砍到 25KB → AI 在几千页里自己找条款 → **漏条款、编造条款号**（这正是 `route.ts` prompt 里反复警告「禁止编造条款号」却防不住的根因）。

> 结论：朋友的诊断完全成立，而且我们项目的情况比他设想的更典型。

## 2. 条款库设计（文档库 → 条款库）

### 2.1 载体选择

**推荐 SQLite**（项目已经在用，`lib/db.ts` 里有 `saveReviewRecordToSqlite`，未登录用户走这条路）。理由：
- 574 条款起步，结构化查询（按构造类型/材料/工艺筛选）正是 SQL 的强项。
- 比 Excel 更适合代码查询；比 Supabase 更轻，本地就能跑通。
- 后续要共享给团队再迁 Supabase，表结构照搬。

### 2.2 字段表结构 `clause`

| 字段 | 类型 | 说明 | 示例 |
|------|------|------|------|
| `id` | INTEGER PK | 自增主键 | 1 |
| `standard_code` | TEXT | 规范号 | `DB44/T 1876-2016` |
| `standard_name` | TEXT | 规范名 | `建筑施工轮扣式钢管模板支撑技术规程` |
| `clause_no` | TEXT | 条款号 | `5.2.3` |
| `clause_title` | TEXT | 条款主题（**匹配用，人工提炼**） | `立杆间距构造要求` |
| `clause_text` | TEXT | 条款原文（**必须完整复制，不增删**） | `立杆间距不应大于 1.2m...` |
| `audit_points` | TEXT | 审核要点（告诉 AI 这条审什么） | `核对方案立杆纵横向间距是否 ≤1.2m` |
| `profession` | TEXT | 专业 id | `template` |
| `structure_type` | TEXT | 构造类型（可空） | `轮扣式` |
| `trigger_materials` | TEXT | 触发材料（逗号串） | `钢管,可调底座,轮盘` |
| `trigger_processes` | TEXT | 触发工艺（逗号串） | `搭设,拆除,监测` |
| `hazard_level` | TEXT | 危大等级（可空） | `危大` |
| `priority` | INTEGER | 优先级 1-5，核心条款=5 | 5 |
| `enabled` | INTEGER | 启用开关 | 1 |
| `created_at` / `updated_at` | TEXT | 时间戳 | — |

> 关键洞察：`structure_type` / `trigger_materials` / `trigger_processes` 这三个字段，正是为了让 **Hook 1 的输出能直接命中条款**。Hook 1 输出 `{构造类型, 涉及材料, 关键工艺}`，Hook 2 拿这三个字段当钥匙去捞条款——这就是"撕书"的机械动作。

### 2.3 怎么填

第一版**手工录入**，先跑通最小闭环：
- 把 DB44/T 1876-2016 的 **5 条核心条款**（朋友说的那 5 条）拆成 5 行塞进去。
- 录入时重点把 `clause_text` 抠完整、`audit_points` 写清楚——这两列直接决定 AI 引用准不准。

## 3. 三个 Hook 详细设计

### Hook 1 · 预处理（替换 `identifyProfessionTypes`）

**位置**：文件解析后、知识库匹配前（`route.ts` 步骤 2~3 之间）。

**职责**：不是关键词打分返回大类，而是抽出**结构化特征**。

```ts
// 输出结构
interface SchemeFeatures {
  profession: string            // 大类（保留，兼容旧逻辑）  "template"
  structureType: string         // 构造细分（新）            "轮扣式"
  materials: string[]           // 涉及材料（新）            ["钢管","可调底座","轮盘"]
  processes: string[]           // 关键工艺（新）            ["搭设","拆除","监测"]
  hazardLevel: string           // 危大判定（新）            "危大"
  possibleStandards: string[]   // 可能违反的规范（新）       ["DB44/T 1876-2016"]
}

async function identifySchemeFeatures(documentContent: string): Promise<SchemeFeatures>
```

**实现选项（决策点）**：
- A. 纯规则：维护一份「构造类型 → 关键词」映射表（轮扣式↔"轮扣/轮盘"，盘扣式↔"盘扣/八孔"）。
- B. 轻量 LLM 调用：让小模型用 structured output 直接吐 JSON（多一次 API，但更准、更通用）。

### Hook 2 · 弹药匹配（新增，串在 `matchRelevantFiles` 之后）

**位置**：`extractKnowledgeContext` 内，现有「按文件夹匹配」之后。

**职责**：根据 Hook 1 的特征，从条款库精准捞条款原文。

```ts
async function matchPreciseClauses(features: SchemeFeatures): Promise<Clause[]>
// 查询逻辑（按优先级）：
//  1. structure_type = features.structureType      → 必捞（priority 5 全要）
//  2. trigger_materials ∩ features.materials       → 捞
//  3. trigger_processes ∩ features.processes       → 捞
//  4. hazard_level = features.hazardLevel          → 捞
// 返回：每条带 {规范号, 条款号, 条款原文, 审核要点}
```

**关键策略：并联，不替换。** 老的 `matchRelevantFiles`（整本规范背景）保留，Hook 2 的精准条款叠在它后面。最终喂给 AI 的上下文：

```
最终上下文 = 整本规范文档（背景）+ 【精准条款锚点】（Hook 2 捞出的条款原文 + 审核要点）
```

→ AI 既有全局背景，又有被高亮、必须查的核心条款。prompt 里「必须完整复制原文，不得编造条款号」这才真正能落地——因为原文现在是代码确定性地塞进去的。

### Hook 3 · 任务拆分（插入 `handleChunkedReviewSSE`）

**位置**：大文档分块后、调 AI 前。

**职责**：把 Hook 2 的条款包按主题切小，每个 chunk 拿到**专属的、干净的**条款子集。

```ts
function assignClausesToChunks(clauses: Clause[], chunks: DocumentChunk[]): Map<chunkId, Clause[]>
//  主题路由（按 clause_title 归类）：
//   材料规格类条款 → 塞给「材料」章节的 chunk
//   构造要求类条款 → 塞给「搭设/构造」章节的 chunk
//   拆除强度类条款 → 塞给「拆除」章节的 chunk
```

→ 每个子任务上下文干净：只审材料规格的子任务，只看到钢管规格条款原文 + 方案材料章节，不被搭设/拆除条款干扰。

## 4. 改造后的完整流程

```
上传文件
   │
[1] 文件解析  parseUploadedFile
   │
[2] Hook 1 预处理  identifySchemeFeatures        ◀── 替换 identifyProfessionTypes
   │      输出：{大类, 构造类型, 材料, 工艺, 危大等级}
   │
[3] 文件匹配  matchRelevantFiles（保留，做背景）
   │
[4] Hook 2 弹药匹配  matchPreciseClauses          ◀── 新增
   │      输出：精准条款包（编号 + 原文 + 审核要点）
   │
[5] 分块判定  shouldUseChunkedReview
   │
   ├─ 小文档 ─► buildReviewPrompt（整本背景 + 精准条款锚点）─► AI
   │
   └─ 大文档 ─► splitDocumentBySections
                 │
                 [6] Hook 3 任务拆分  assignClausesToChunks   ◀── 新增
                 │      每个 chunk 拿专属条款子集
                 │
                 每个 chunk ─► AI 子任务（上下文干净）
                 │
                 buildMergePrompt 智能汇总 ─► 最终报告
   │
[7] 保存 + SSE 返回
```

## 5. 分步实施路线图（每步都可独立验证）

> 设计原则：**先并联、后替换；先跑通最小闭环，再铺开。** 每一步都给出「怎么验证成功」。

| 步骤 | 动作 | 验证标准 |
|------|------|---------|
| **Step 0** | 建条款库表 + 手工录入 DB44/T 1876 那 5 条 | `SELECT * FROM clause` 能看到 5 行，原文完整 |
| **Step 1** | 写 `matchPreciseClauses`，features 先硬编码 `{structureType:"轮扣式"}` 调用 | 控制台打印出捞到的 5 条条款，编号+原文对得上 |
| **Step 2** | 写 `identifySchemeFeatures`（先按决策点选 A 或 B） | 喂一份真实轮扣式方案，输出 JSON 的 `structureType` 是"轮扣式" |
| **Step 3** | 把 Hook 1 + Hook 2 串进 `extractKnowledgeContext`；prompt 增加「精准条款锚点」段 | 上传轮扣式方案，AI 引用的就是那 5 条，**条款号不再编造** |
| **Step 4** | Hook 3 接进 `handleChunkedReviewSSE` | 大文档分块时，每个 chunk 的 prompt 里能看到对应主题的条款 |

> Step 3 是整个改造的「价值验证点」——如果 AI 引用条款的准确率明显上升，方向就对了，再投资 Step 4 和扩条款库。

## 6. 需要你和朋友拍板的决策点

1. **条款库载体**：SQLite（推荐，项目已有）/ Excel / Supabase？
2. **Hook 1 实现**：纯规则映射表（省一次 API、可控）vs 轻量 LLM structured output（更准、多一次调用）？
3. **Hook 2 与老逻辑关系**：并联保留整本规范做背景（推荐）vs 直接替换？
4. **条款库初期范围**：只录模板专业（轮扣式起步）跑通 vs 一开始就全专业铺开？
5. **触发字段存储**：逗号串 + `LIKE` 查询（简单）vs 关联表（规范但重）？

## 7. 风险提示

- `clause_text` 必须由懂规范的人录入，**原文一字不改**——否则就是把"AI 编造"换成"人工录入错误"，换汤不换药。
- 条款库覆盖率决定上限：前期只录 5 条时，AI 只在这 5 条上变准，其它条款仍是老问题。这是预期的、可接受的渐近路径。
- Hook 1 若选 LLM 方案，会多一次 API 调用和延迟，需评估审核时长的可接受度。
